# ✅ **REACT INTERVIEW QUICK-REVISION NOTES**

*(Foundations → Traps → One-liners)*

---

## 1️⃣ What is React

**Best answer:**

> React is a JavaScript library for building user interfaces using reusable components and state-driven rendering.

**Key idea:**

- React describes **what UI should look like**

- React handles **updating the UI**

❌ Don’t call it a framework  
❌ Don’t say “only for SPA”

---

## 2️⃣ What problem does React solve

**Answer:**

> React solves the problem of managing complex and frequently changing UIs by automatically keeping the UI in sync with state.

**Without React:**

- Manual DOM updates

- Hard to maintain state

---

## 3️⃣ Component

**Definition:**

> A component is a reusable, independent piece of UI that encapsulates structure, logic, and behavior.

**Mental model:**

> Component = function that returns UI for a given state

---

## 4️⃣ Types of Components

- Functional components

- Class components (legacy)

👉 Functional components are preferred today.

---

## 5️⃣ Why Functional Components

**Key reasons:**

- Simpler and more readable

- Work naturally with hooks

- Easier to reuse logic

- Easier to test

❌ Don’t say “class components are bad”

---

## 6️⃣ State

**Definition:**

> State is data managed by a component that determines how it renders and behaves over time.

**Rule:**

> State change → re-render

**Mental model:**

> State = component’s memory

---

## 7️⃣ State vs Props

| State              | Props              |
| ------------------ | ------------------ |
| Owned by component | Passed from parent |
| Mutable via setter | Read-only          |
| Internal data      | External input     |

---

## 8️⃣ Updating State (Functional Component)

**Correct way:**

> Use the setter function returned by `useState`.

❌ Don’t say `setState` (that’s class component)

---

## 9️⃣ Functional State Update

**Why needed:**

> Because state updates are asynchronous and may be batched.

**Rule:**

> If new state depends on previous state → use functional updater

---

## 🔟 Re-Rendering

A component re-renders when:

1. Its **state changes**

2. Its **props change**

3. Its **parent re-renders**

❌ Not when variables change  
❌ Not when `render()` is called manually

---

## 1️⃣1️⃣ Parent → Child Re-Render

**Default behavior:**

> When a parent re-renders, child components also re-render.

**Exception:**

> Memoized children (`React.memo`) may skip re-render.

---

## 1️⃣2️⃣ React.memo

**Definition:**

> `React.memo` memoizes a component and skips re-rendering if props have not changed.

**Important:**

- Uses **shallow comparison**

- Use only for expensive components

---

## 1️⃣3️⃣ useCallback

**Purpose:**

> Memoizes a function reference to prevent unnecessary re-renders.

**Key rule:**

> `useCallback` memoizes function identity, not function result.

Used when:

- Passing functions to memoized children

- Using functions as dependencies

---

## 1️⃣4️⃣ useMemo

**Purpose:**

> Memoizes the result of an expensive computation.

**Difference:**

- `useCallback` → function

- `useMemo` → value

---

## 1️⃣5️⃣ useEffect

**Definition:**

> `useEffect` is used to perform side effects **after rendering**.

Examples:

- API calls

- subscriptions

- timers

- DOM interactions

❌ Does NOT run during render

---

## 1️⃣6️⃣ useEffect Dependency Array

### 3 cases:

1️⃣ No array → runs after every render  
2️⃣ Empty array `[]` → runs once after mount  
3️⃣ `[deps]` → runs when any dependency changes

---

## 1️⃣7️⃣ useEffect Cleanup

**Definition:**

> Cleanup function removes side effects created by the effect.

**Runs when:**

- Before effect re-runs

- On component unmount

**Mental model:**

> Effect sets up → cleanup tears down

---

## 1️⃣8️⃣ Lists & Keys

**Why keys exist:**

> Keys help React identify which list items have changed, been added, or removed.

**Rule:**

> Keys must be unique and stable among siblings.

---

## 1️⃣9️⃣ Array Index as Key

**Acceptable only when:**

- List is static

- No reordering, adding, or removing

❌ Bad for dynamic lists  
❌ Causes UI & state bugs

---

## 2️⃣0️⃣ Lifting State Up

**Definition:**

> Moving shared state to the nearest common parent so multiple components stay in sync.

**Why:**

- Single source of truth

- Avoid inconsistent UI

---

## 2️⃣1️⃣ Data Flow in React

**Golden rule:**

> Data flows one-way in React — from parent to child.

Props & context do **not** change direction.

---

## 2️⃣2️⃣ Context

**Problem solved:**

> Prop drilling.

**Use for:**

- Theme

- Auth user

- Global config

❌ Not for frequent updates  
❌ Not replacement for state management

---

## 2️⃣3️⃣ When to Avoid Context

Avoid when:

- Data needed only by few components

- Props can be passed easily

- Updates are very frequent

---

## 2️⃣4️⃣ Controlled vs Uncontrolled Components

### Controlled:

- React state is source of truth

- Uses `value` + `onChange`

### Uncontrolled:

- DOM is source of truth

- Uses refs

**Mental model:**

> Controlled → React controls data  
> Uncontrolled → DOM controls data

---

## 🔑 **REACT INTERVIEW FORMULAS (MEMORIZE)**

- State change → re-render

- Parent re-render → children re-render (default)

- Same props + expensive render → `React.memo`

- Memoize function → `useCallback`

- Memoize value → `useMemo`

- Side effects → `useEffect`

- Shared state → lift up

- Many prop levels → Context
