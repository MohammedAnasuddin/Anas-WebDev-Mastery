# ✅ **DATABASES — POSTGRESQL & MONGODB (INTERVIEW NOTES)**

---

## 1️⃣ What is a Database

**Best answer:**

> A database is an organized system for storing and managing data persistently, allowing efficient querying, consistency, and reliability.

**Why needed:**

- Persistence (survives restarts)

- Structured storage

- Efficient querying

- Data integrity & concurrency

---

## 2️⃣ SQL vs NoSQL

| SQL                  | NoSQL                       |
| -------------------- | --------------------------- |
| Relational           | Non-relational              |
| Fixed schema         | Flexible schema             |
| Tables               | Documents / collections     |
| Strong relationships | Embedded or loose relations |

**Rule:**

> SQL → structured & relational  
> NoSQL → flexible & scalable

---

## 3️⃣ What is PostgreSQL

**Answer:**

> PostgreSQL is an open-source relational database management system (RDBMS) that uses SQL.

**Known for:**

- Reliability

- Strong consistency

- Advanced features (indexes, constraints, JSONB)

---

## 4️⃣ SQL Command Types

- **DDL (Data Definition Language)**  
  Structure → `CREATE`, `ALTER`, `DROP`

- **DML (Data Manipulation Language)**  
  Data change → `INSERT`, `UPDATE`, `DELETE`

- **DQL (Data Query Language)**  
  Data read → `SELECT`

**Formula:**

> DDL = structure  
> DML = data change  
> DQL = data read

---

## 5️⃣ Table & Row

- **Table** → structured data organized into rows and columns

- **Row** → a single record/entity in a table

---

## 6️⃣ Primary Key

**Definition:**

> A primary key uniquely identifies each row in a table.

**Rules:**

- Unique

- Cannot be NULL

- One logical primary key per table

---

## 7️⃣ Foreign Key

**Definition:**

> A foreign key is a column that references the primary key of another table.

**Purpose:**

- Enforces relationships

- Maintains referential integrity

**Rule:**

> Primary key = identity  
> Foreign key = relationship

---

## 8️⃣ Relationships in SQL

1️⃣ One-to-One  
2️⃣ One-to-Many  
3️⃣ Many-to-Many (via junction table)

---

## 9️⃣ JOIN

**Definition:**

> JOIN combines rows from two or more tables based on a related column.

**Purpose:**

- Fetch related data across tables

---

## 🔟 Types of JOINs

- **INNER JOIN**

- **LEFT JOIN (LEFT OUTER JOIN)**

- **RIGHT JOIN (RIGHT OUTER JOIN)**

- **FULL JOIN (FULL OUTER JOIN)**

- **CROSS JOIN**

**Memory order:**

> INNER, LEFT, RIGHT, FULL, CROSS

---

## 1️⃣1️⃣ INNER vs LEFT JOIN

- **INNER JOIN** → only matching rows from both tables

- **LEFT JOIN** → all rows from left + matching rows from right

If no match in LEFT JOIN → right side is **NULL**

---

## 1️⃣2️⃣ Index

**Definition:**

> An index improves query performance by allowing faster data lookup.

**Trade-off:**

- Faster reads

- Slightly slower writes

**Rule:**

> Index improves SELECT, not INSERT/UPDATE

---

## 1️⃣3️⃣ Constraints

Common constraints:

- `PRIMARY KEY`

- `FOREIGN KEY`

- `UNIQUE`

- `NOT NULL`

- `CHECK`

**Purpose:**

> Enforce data integrity

---

## 1️⃣4️⃣ Transaction

**Definition:**

> A transaction is a sequence of operations executed as a single unit.

---

## 1️⃣5️⃣ ACID Properties

- **Atomicity** → all or nothing

- **Consistency** → valid state before & after

- **Isolation** → transactions don’t interfere

- **Durability** → committed data persists

---

# 🍃 **MONGODB BASICS**

---

## 1️⃣ What is MongoDB

**Answer:**

> MongoDB is a NoSQL document-based database that stores data in flexible JSON-like documents.

---

## 2️⃣ Collection & Document

- **Collection** → similar to a table

- **Document** → similar to a row (JSON-like)

---

## 3️⃣ Schema in MongoDB

- Flexible schema

- Fields can vary between documents

**Rule:**

> Schema flexibility ≠ no structure

---

## 4️⃣ `_id`

- Default primary key in MongoDB

- Unique identifier for each document

---

## 5️⃣ Relationships in MongoDB

Two approaches:  
1️⃣ **Embedding** (preferred for related data)  
2️⃣ **Referencing** (store `_id` of another document)

---

## 6️⃣ When to Use MongoDB vs PostgreSQL

**Use PostgreSQL when:**

- Strong relationships

- Transactions

- Data integrity is critical

**Use MongoDB when:**

- Flexible data

- Rapid schema changes

- High write throughput

---

## 🔑 **DATABASE INTERVIEW FORMULAS (MEMORIZE)**

- SQL → structured & relational

- NoSQL → flexible & scalable

- Primary key → identity

- Foreign key → relationship

- INNER JOIN → matching rows

- LEFT JOIN → left preserved

- Index → faster reads

- Transaction → ACID

---

## ⚠️ **COMMON INTERVIEW TRAPS**

❌ PostgreSQL is not NoSQL  
❌ DML ≠ Data Markup Language  
❌ JOIN does not modify data  
❌ Index does not speed up writes  
❌ MongoDB has schema (but flexible)

---

# 🍃 **MONGODB — INTERVIEW QUICK NOTES**

---

## 1️⃣ What is MongoDB

**Best answer:**

> MongoDB is a NoSQL, document-oriented database that stores data in flexible, JSON-like documents.

**Key ideas:**

- NoSQL database

- Document-based

- Flexible schema

---

## 2️⃣ Collection & Document

- **Collection** → group of documents

- **Document** → single record stored in BSON (JSON-like)

**Mental model:**

> Collection ≈ table  
> Document ≈ row (conceptually)

---

## 3️⃣ `_id` Field

**Definition:**

> `_id` is a unique identifier that uniquely identifies each document in a collection.

**Important points:**

- Mandatory

- Unique

- Auto-generated if not provided

- Acts like a primary key

---

## 4️⃣ Schema in MongoDB

**Truth (interview-safe):**

> MongoDB has a schema, but it is flexible and not enforced by default.

**Clarification:**

- Documents in the same collection can have different fields

- Schema validation **can** be enforced if needed

---

## 5️⃣ Relationships in MongoDB

MongoDB does **not** use joins like SQL by default.

Two approaches:

### 1️⃣ Embedding

> Store related data inside the same document

- Faster reads

- Atomic updates

- Best for small, tightly related data

### 2️⃣ Referencing

> Store `_id` of another document

- Avoids duplication

- Better for large or shared data

- Similar to foreign keys (conceptually)

---

## 6️⃣ When to Embed vs Reference

**Rule:**

> Read together → embed  
> Grow or share → reference

---

## 7️⃣ Querying in MongoDB

**Basic querying:**

> Documents are queried using methods like `.find()` with filters.

**Mental model:**

> Query = find documents matching criteria

---

## 8️⃣ Filter vs Projection

- **Filter** → decides *which documents* are returned

- **Projection** → decides *which fields* are returned

**One-line formula:**

> Filter = rows  
> Projection = columns

---

## 9️⃣ Indexes

**Definition:**

> An index is a data structure that improves query performance by allowing faster lookup of documents.

**Trade-off:**

- Faster reads

- Slower writes

- Extra storage

**Rule:**

> Index speeds up queries, not inserts

---

## 🔟 MongoDB vs PostgreSQL

**Use MongoDB when:**

- Schema is flexible or evolving

- Data is semi-structured

- High write throughput

- Horizontal scalability needed

**Use PostgreSQL when:**

- Strong relationships

- Complex joins

- Transactions & constraints

- Strong consistency needed

---

## 🔑 **MONGODB INTERVIEW FORMULAS (MEMORIZE)**

- MongoDB = NoSQL + document-based

- `_id` = primary key

- Schema-flexible ≠ schema-less

- Embed → fast reads

- Reference → scalable relationships

- Filter → documents

- Projection → fields

- Index → faster queries

---

❌ “MongoDB is cloud-only”  
❌ “MongoDB replaces SQL”  
❌ “Indexes improve writes”  
❌ “MongoDB uses SQL joins by default”
